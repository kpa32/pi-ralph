
## 快速开始
### 工作

1. 接收用户的功能描述
2. 提出 3-5 个重要的澄清问题（带有字母选项）
3. 识别“待完成工作”（JTBD），并将其分解为“关注主题”。
每个主题对应一个规格说明文件，放在 `.ralph/specs/` 目录下。

**重要：** 不要开始实施。只需创建 规格说明文件 即可。

## 第一步: 澄清问题


## 第 1 步：澄清问题

仅在初始提示不明确的情况下提出关键问题。重点关注：

- **问题/目标：** 这解决了什么问题？
- **核心功能：** 有哪些关键举措？
- **范围/边界：** 它不应该做什么？
- **成功标准：** 我们怎么知道它已经完成了？

### 像这样格式化问题：

````
1. 此功能的主要目标是什么？
   A. 改善用户入门体验
   B. 提高用户保留率
   C. 减少支持负担
D.其他：[请注明]

2. 目标用户是谁？
   A. 仅限新用户
   B. 仅限现有用户
   C、所有用户
D. 仅限管理员用户

3. 范围是什么？
   A. 最小可行版本
   B. 全功能实现
   C. 只是后端/API
D. 只是用户界面
````

这让用户可以用“1A、2C、3B”进行响应，以便快速迭代。请记住缩进选项。

## 第2步：规格说明文件结构

使用以下部分生成 规格说明文件：

### 1. 简介/概述
简要描述该功能及其解决的问题。

### 2. 目标
具体的、可衡量的目标（项目符号列表）。

### 3. 用户故事
每个故事都需要：
- **标题：** 简短的描述性名称
- **说明：** “作为[用户]，我想要[功能]以便[受益]”
- **验收标准：** “完成”含义的可验证清单

每个故事都应该足够小，以便在一次重点会议中实施。

**格式：**
``降价
### US-001：[标题]
**说明：** 作为[用户]，我想要[功能]以便[受益]。

**验收标准：**
- [ ] 具体可验证标准
- [ ] 另一个标准
- [ ] 类型检查/lint 通过
- [ ] **[仅限 UI 故事]** 使用支持视觉元素技能拓展(浏览器,窗口截图,用户同提供的验证程序)进行验证
````

**重要：** 
- 验收标准必须是可验证的，而不是模糊的。 “正常工作”是不好的。 “删除前按钮显示确认对话框”很好。
- **对于任何 UI 发生更改的故事：** 始终将“视觉元素技能拓展”作为接受标准。这确保了前端工作的视觉验证。

### 4. 功能需求
具体功能的编号列表：
- “FR-1：系统必须允许用户...”
- “FR-2：当用户单击 X 时，系统必须...”

明确且明确。

### 5. 非目标（超出范围）
此功能不包括什么。对于管理范围至关重要。

### 6. 设计考虑因素（可选）
- 用户界面/用户体验要求
- 链接到模型（如果有）
- 要重用的相关现有组件

### 7. 技术考虑（可选）
- 已知的约束或依赖性
- 与现有系统的集成点
- 性能要求

### 8. 成功指标
如何衡量成功？
- “完成 X 的时间减少 50%”
- “转化率提高10%”

### 9. 开放式问题
其余问题或需要澄清的领域。

## 为初级开发人员编写

specs/*规格说明文件.md 读者可能是初级开发人员或 AI 代理。因此：

- 明确且不含糊
- 避免使用行话或解释它
- 提供足够的细节以理解目的和核心逻辑
- 数量要求，方便参考
- 使用有帮助的具体例子

---

## 输出

- **格式：** 降价（`.md`）
- **地点：** `.ralph/specs/`
- **文件名：** `[number]-[feature-name].md` （烤肉串盒）

## 示例 spec

``降价
# spec：任务优先级系统

## 简介

为任务添加优先级，以便用户可以专注于最重要的事情。任务可以标记为高、中或低优先级，并具有可视化指示器和过滤功能，帮助用户有效管理其工作负载。

## 目标

- 允许为任何任务分配优先级（高/中/低）
- 在优先级之间提供清晰的视觉区分
- 启用按优先级过滤和排序
- 默认新任务为中等优先级

## 用户故事

### US-001：向数据库添加优先级字段
**说明：** 作为一名开发人员，我需要存储任务优先级，以便它在会话中保持不变。

**验收标准：**
- [ ] 将优先级列添加到任务表：'high' | '中等' | '低'（默认'中'）
- [ ] 成功生成并运行迁移
- [ ] 类型检查通过

### US-002：在任务卡上显示优先级指示器
**说明：** 作为用户，我希望一目了然地看到任务优先级，这样我就知道首先需要注意什么。

**验收标准：**
- [ ] 每个任务卡都显示彩色优先级徽章（红色=高，黄色=中，灰色=低）
- [ ] 无需悬停或单击即可看到优先级
- [ ] 类型检查通过
- [ ] 使用开发浏览器技能在浏览器中验证

### US-003：将优先级选择器添加到任务编辑中
**说明：** 作为用户，我想在编辑任务时更改任务的优先级。

**验收标准：**
- [ ] 任务编辑模式中的优先级下拉菜单
- [ ] 显示当前选定的优先级
- [ ] 选择更改后立即保存
- [ ] 类型检查通过
- [ ] 使用开发浏览器技能在浏览器中验证

### US-004：按优先级过滤任务
**说明：** 作为用户，我想过滤任务列表，以便在我集中注意力时仅查看高优先级项目。

**验收标准：**
- [ ] 过滤器下拉菜单选项：全部 |高|中等|低
- [ ] 过滤器保留在 URL 参数中
- [ ] 当没有任务匹配过滤器时显示空状态消息
- [ ] 类型检查通过
- [ ] 使用开发浏览器技能在浏览器中验证

## 功能需求

- FR-1：添加 `priority` 任务表字段（'高'|'中'|'低'，默认'中'）
- FR-2：在每个任务卡上显示彩色优先级徽章
- FR-3：在任务编辑模式中包含优先级选择器
- FR-4：将优先级过滤器下拉列表添加到任务列表标题
- FR-5：按每个状态列中的优先级排序（高到中到低）

## 非目标

- 没有基于优先级的通知或提醒
- 没有根据截止日期自动分配优先级
- 子任务没有优先级继承

## 技术考虑

- 重复使用具有颜色变体的现有徽章组件
- 通过 URL 搜索参数管理过滤器状态
- 优先级存储在数据库中，而不是计算

## 成功指标

- 用户只需点击 2 次即可更改优先级
- 高优先级任务立即显示在列表顶部
- 任务列表性能没有下降

## 开放性问题

- 优先级是否应该影响列内的任务排序？
- 我们应该添加键盘快捷键来更改优先级吗？
````
