# 需求讨论与规格说明书生成

## 目标
通过与用户对话，识别 **Jobs to Be Done (JTBD)**，并将其分解为 **Topics of Concern**，最终生成规格说明书。

## 工作流程

### 1. 对话引导
- 接收用户的功能描述或产品想法
- 引导对话识别核心 **Jobs to Be Done (JTBD)** - 用户想要完成什么工作
- 对于每个 JTBD，识别相关的 **Topics of Concern**（关注点）

### 2. 澄清问题
提出 3-5 个关键澄清问题（带有字母选项），重点关注：
- **问题/目标**：这个功能解决什么用户问题？
- **目标用户**：谁是主要使用者？
- **范围边界**：应该包含什么，不应该包含什么？
- **成功标准**：如何判断功能成功？

**问题格式示例**：
```
1. 这个功能的主要目标是什么？
   A. 改善特定用户群体的体验
   B. 解决一个具体的用户痛点
   C. 实现一个商业目标
   D. 其他：[请说明]

2. 目标用户是谁？
   A. 新用户
   B. 现有用户
   C. 管理员/运营人员
   D. 所有用户

3. 功能的初始范围应该是？
   A. 最小可行版本（MVP）
   B. 完整功能实现
   C. 仅核心流程
   D. 带有扩展点的基础版本
```

### 3. 识别 Jobs to Be Done (JTBD) 与 Topics of Concern
基于对话内容，系统性地识别以下要素：

1. **Jobs to Be Done (JTBD) - 待完成工作**
   - 定义用户需要达成的目标或结果，而非具体的功能特性
   - 关注用户希望完成什么"工作"，而非如何完成
   - 示例：
     - ✓ "帮助设计师快速创建情绪板"
     - ✓ "让团队成员高效协作评审设计"

2. **Topics of Concern - 关注点**
   - 将每个 JTBD 分解为相互独立、内聚的关注点
   - **关注点粒度测试：单句无"和"测试 (One Sentence Without 'And' Test)**
     - 判断标准：能否在不使用"和"连接多个不相关能力的情况下，用一句话清晰描述该关注点？
     - ✓ **通过示例**："颜色提取系统分析图片以识别主要颜色"
     - ✗ **不通过示例**："用户系统处理身份验证、个人资料和计费" → 这实际上是3个独立的关注点
   - 如果描述中需要使用"和"，则表明该关注点包含多个应独立处理的组件
   - 示例：JTBD "创建情绪板" 可分解为以下关注点：
     - 图片上传与收集
     - 颜色提取与分析  
     - 布局排列与展示
     - 分享与协作功能

3. **关系映射**
   - 1个 JTBD → 多个 Topics of Concern
   - 1个 Topic of Concern → 1个规格说明书（Spec）
   - 1个规格说明书 → 多个实现任务（Tasks）

### 4. 生成规格说明书
每个 **Topic of Concern** 对应一个规格说明书文件，保存在 `.ralph/specs/` 目录。

**文件名格式**：`[编号]-[topic-name].md`（使用烤肉串命名法）

---

## 规格说明书结构

每个规格说明书应包含以下部分：

### 1. 概述
- **关联的 JTBD**：这个 topic 服务于哪个 JTBD
- **Topic of Concern**：本文件关注的具体方面
- **目标**：简要描述要达成的目标

### 2. 用户故事（可选但推荐）
使用 User Story 格式描述具体功能需求：
```
### US-001：[标题]
**描述**：作为[角色]，我想要[功能]，以便[价值]。

**验收标准**：
- [ ] 具体可验证的标准1
- [ ] 具体可验证的标准2
- [ ] 类型检查通过
- [ ] 测试通过（如适用）
- [ ] **[UI相关]** 使用视觉验证工具验证
```

**用户故事粒度原则**：
- 每个故事应小到能在一个上下文窗口内完成
- 避免使用"和"连接多个独立功能
- 验收标准必须是可验证的，而非模糊描述

### 3. 功能需求
按优先级排序的具体功能列表：
- FR-1：系统必须能够...
- FR-2：当用户执行 X 时，系统应该...
- FR-3：性能要求：响应时间 < 100ms

### 4. 验收标准
从用户故事中提取的综合性验收标准，确保功能完整可用。

### 5. 非目标（范围边界）
明确说明本规格说明书**不包含**的内容，防止范围蔓延。

### 6. 设计考虑（如适用）
- UI/UX 要求
- 需要重用的现有组件
- 品牌/设计系统约束

### 7. 技术考虑（如适用）
- 技术栈限制
- 性能要求
- 集成点
- 数据存储需求

### 8. 成功指标
如何衡量这个 Topic 的成功：
- 用户体验指标（完成时间、错误率）
- 业务指标（使用率、满意度）
- 技术指标（性能、稳定性）

### 9. 开放问题
需要进一步讨论或研究的领域。

## 写作指南

### 为 AI 代理和开发人员编写
- **明确具体**：避免模糊表述，使用可验证的描述
- **技术中立**：描述"做什么"，而非"如何做"
- **完整一致**：确保需求间没有矛盾
- **适度详细**：提供足够细节，但不规定实现细节

### 验收标准原则
- ✅ **好的**："上传图片后 3 秒内显示预览"
- ❌ **不好的**："上传功能工作正常"
- ✅ **好的**："错误发生时显示具体错误信息"
- ❌ **不好的**："良好的错误处理"

### 视觉验证要求
对于 UI 相关的功能，必须包含视觉验证要求：
- 使用浏览器开发工具验证
- 确保响应式设计正常工作
- 验证交互反馈符合预期

## 示例：情绪板应用的规格说明书

### 文件：`.ralph/specs/01-image-upload.md`

```markdown
# 图片上传与收集系统

## 概述
- **关联的 JTBD**：帮助设计师创建情绪板
- **Topic of Concern**：图片收集与管理
- **目标**：让设计师能够方便地上传、管理和组织灵感图片

## 用户故事

### US-001：单张图片上传
**描述**：作为设计师，我想要上传单张图片到情绪板，以便收集视觉灵感。

**验收标准**：
- [ ] 支持拖拽上传和文件选择器上传
- [ ] 支持 JPG、PNG、GIF、WebP 格式
- [ ] 文件大小限制：最大 10MB
- [ ] 上传时显示进度指示
- [ ] 上传成功后在情绪板中显示缩略图
- [ ] 类型检查通过
- [ ] 使用视觉验证工具验证上传流程

### US-002：批量图片上传
**描述**：作为设计师，我想要一次上传多张图片，提高收集效率。

**验收标准**：
- [ ] 支持多文件选择
- [ ] 批量上传时显示总体进度和单个文件状态
- [ ] 单个文件失败不影响其他文件上传
- [ ] 上传完成后显示成功/失败统计
- [ ] 类型检查通过
- [ ] 视觉验证批量上传界面

## 功能需求
- FR-1：图片上传组件，支持拖拽和点击选择
- FR-2：文件类型验证和大小限制
- FR-3：上传进度反馈
- FR-4：错误处理（网络错误、格式错误、大小超限）
- FR-5：上传后的图片预览和基本信息显示

## 验收标准
- 上传成功率 > 99%
- 大文件（8-10MB）上传时间 < 10秒（在 50Mbps 网络下）
- 移动端和桌面端体验一致

## 非目标
- 不支持视频上传
- 不支持图片编辑（裁剪、滤镜等）
- 不支持云端图片库集成（后续版本考虑）

## 设计考虑
- 使用现有设计系统的上传组件
- 保持与现有情绪板布局的一致性
- 移动端优化：触控友好的交互

## 技术考虑
- 使用现有文件上传服务
- 图片存储使用现有 CDN
- 需要生成不同尺寸的缩略图

## 成功指标
- 图片上传放弃率 < 5%
- 用户满意度评分 > 4/5
- 平均上传时间 < 5秒

## 开放问题
- 是否支持从 URL 导入图片？
- 是否需要图片去重功能？
```

## 质量检查清单

在保存规格说明书前，确认：

- [ ] 已识别清晰的 JTBD
- [ ] Topics of Concern 粒度合适（一句话不用"和"）
- [ ] 用户故事足够小（一个迭代可完成）
- [ ] 验收标准可验证、不模糊
- [ ] 非目标明确，防止范围蔓延
- [ ] 文件名符合规范：`[编号]-[topic-name].md`
- [ ] 保存到正确目录：`.ralph/specs/`

## 核心概念

### Jobs to Be Done (JTBD)
用户想要完成的"工作"或目标，关注结果而非功能。例如："快速完成设计评审"而非"创建评论功能"。

### Topics of Concern
JTBD 的具体组成部分，每个 Topic 关注一个独立的方面。好的 Topic 应该是内聚的、可独立实现的。

### 规格说明书 (Specifications)
描述每个 Topic of Concern 的详细需求文档，作为开发的唯一事实源。

### Ralph 工作流集成
生成的规格说明书将用于：
1. **计划阶段**：对比 specs 与现有代码，生成 BUILD_PLAN.md
2. **构建阶段**：根据 BUILD_PLAN.md 实现功能
3. **验证阶段**：根据验收标准验证实现质量

**记住**：规格说明书是开发的起点，不是终点。在实现过程中可能会发现需要调整的地方，及时更新规格说明书以保持同步。
